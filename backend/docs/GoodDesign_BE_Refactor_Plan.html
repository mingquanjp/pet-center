Good Design Backend Refactor Plan - Pet Center

Tài liệu dành cho thành viên triển khai refactor backend theo High Cohesion, Low Coupling, SOLID và Design Patterns.

Backend: Express 5 + TypeScript Database: PostgreSQL/pg Scope: backend/src Không đổi API contract nếu không có yêu cầu riêng

1. Tóm tắt kết luận

Backend đã có nền tảng kiến trúc tương đối tốt: chia theo domain module, có route/controller/service/repository/schema/types, có middleware validate, auth, role, error handler và response helper. Vì vậy refactor không nên viết lại toàn bộ. Việc cần làm là tách những file đang quá lớn, giảm phụ thuộc trực tiếp giữa service với notification/mail/database detail, đưa mapping DTO ra khỏi repository, chuẩn hóa controller và thay các nhánh trạng thái lặp lại bằng policy/strategy nhỏ.

Ưu tiên cao nhất: refactor module boarding, sau đó grooming, appointments, pets, reports, invoices. Các module nhỏ hơn chỉ chỉnh những điểm lệch chuẩn rõ ràng, tránh tạo abstraction không cần thiết.

Kết quả mong muốn

  • Mỗi file service chỉ phụ trách một nhóm use case gần nhau, không còn service 1.000+ dòng.
  • Controller chỉ đọc request, gọi service, trả response thống nhất.
  • Repository chỉ truy vấn database và trả row/data thô, không map DTO UI/API phức tạp.
  • Business rule về status/price/availability được tách thành policy/strategy/testable function.
  • Notification/email/hash/token được gọi qua service nhỏ hoặc port rõ trách nhiệm, giảm coupling.
  • Không đổi URL, response shape, schema validate, quyền truy cập và nghiệp vụ đang chạy.

2. Giả định và phạm vi

Giả định

  • Thành viên refactor backend sẽ làm trên code hiện tại trong backend/src.
  • Refactor phục vụ chương Good Design, không thêm nghiệp vụ mới.
  • API đang dùng bởi frontend không được đổi endpoint hoặc response nếu chưa thống nhất với frontend.
  • Database schema không đổi, trừ khi có migration riêng được duyệt.
  • Ưu tiên thay đổi có thể kiểm chứng bằng pnpm --filter backend lint và test hiện có.

Không nằm trong phạm vi refactor này

  • Không đổi framework backend.
  • Không chuyển sang ORM.
  • Không triển khai full DI container nếu chỉ để phục vụ một vài dependency đơn giản.
  • Không gom toàn bộ CRUD admin vào generic base service ngay ở bước đầu, vì có thể over-engineering.

3. Kiến trúc hiện tại

Quy ước kiến trúc đã có trong backend/docs/PetCenter_BE_Express_Architecture_Guide.md:

Route       - khai báo endpoint và gắn middleware
Controller  - nhận request, trả response
Service     - xử lý nghiệp vụ
Repository  - truy vấn database
Schema      - validate params/query/body
Types       - type riêng của module

Quy tắc cần giữ:

Route không xử lý nghiệp vụ.
Controller không viết SQL.
Service không biết Express response.
Repository không trả HTTP response.
Code hiện tại đã đi đúng hướng ở nhiều điểm: app.ts gắn middleware tập trung, routes/index.ts mount module route, hầu hết route dùng validateRequest, service không nhận Response, repository là nơi chủ yếu gọi query.

4.2. Import ở giữa file

boarding.service.ts và boarding.controller.ts có import type ở giữa file sau phần owner function. Đây là dấu hiệu file đã được nối thêm theo từng đợt thay vì chia module. Nên tách file hoặc gom import lên đầu.

4.3. Repository trả DTO API

Các file có dấu hiệu mapper DTO nằm trong repository:

  • pets.repository.ts: mapPet, mapStaffPet, mapPetDetail, mapPetMedicalExam, mapPetSpaHistory.
  • grooming.repository.ts: mapBookingPet, mapGroomingServices, mapGroomingTicketListItem, mapStaffGroomingTicket.
  • dashboard.repository.ts: nhiều mapper owner/staff/doctor/admin dashboard.

Điều này làm database layer phụ thuộc vào API contract. Khi frontend đổi response field, repository phải đổi dù SQL không đổi. Đây là coupling không cần thiết.

4.4. Service phụ thuộc trực tiếp vào notification/mail

Các service gọi trực tiếp event/email:

  • boarding.service.ts: notifyBoardingCreated, notifyBoardingConfirmed, notifyBoardingRejected, notifyBoardingCancelled, notifyBoardingCheckedIn, notifyBoardingUpdateCreated, mailService.sendOwnerAccountCreatedEmail.
  • grooming.service.ts: notifyGroomingCreated, notifyGroomingAccepted, notifyGroomingCompleted.
  • appointments.service.ts: notifyAppointmentConfirmed, notifyAppointmentRejected, notifyMedicalExamCompleted.
  • invoices.service.ts: notifyPaymentSuccess.
  • auth.service.ts: sendPasswordResetEmail.

Đây không sai về chức năng, nhưng coupling cao. Service nghiệp vụ đang phải biết module notification/mail cụ thể. Nên tách thành event publisher/notification port nhỏ để dễ test và dễ thay đổi.

4.5. any làm yếu contract

Các ví dụ cần ưu tiên thay bằng DTO/type:

  • boarding.service.ts: mapper staff/admin nhận any, mapDbBoardingStatusToDto trả any.
  • boarding.repository.ts: admin room payload/filter dùng any.
  • appointments.service.ts: filters any, status/type cast as any.
  • invoices.service.ts: filters any.
  • appointments.controller.ts: dùng (req as any).user.userId dù đã có express auth type.

4.6. Error flow chưa thống nhất

  • Một số controller tự try/catch(next) dù route đã có thể dùng asyncHandler: reports.controller.ts, medical-records.controller.ts.
  • Một số repository ném new Error("GROOMING_SLOT_FULL"), service bắt theo message string.
  • notifications.service.ts ném new Error("Notification not found or access denied"), không qua AppError.

5. Mapping sang Good Design

Nguyên lý Vấn đề trong code hiện tại Refactor chứng minh được
High Cohesion Service/repository lớn chứa nhiều nhóm nghiệp vụ khác nhau. Tách theo subdomain/use case: owner, staff, admin, mapper, policy, notification.
Low Coupling Service gọi trực tiếp notification/mail; repository biết DTO API. Service gọi event publisher; repository trả row; mapper nằm riêng.
SRP boarding.service.ts vừa xử lý lưu trú, vừa tạo owner account, vừa gửi mail, vừa map timeline. Mỗi service/helper có một lý do thay đổi.
OCP Logic status/price/payment dùng if/switch rải rác; thêm status mới phải sửa nhiều chỗ. Tách status transition policy, pricing strategy, report/export strategy.
LSP Hiện chưa có hierarchy lớn, nên không ép tạo class kế thừa. Nếu dùng strategy/interface, mọi implementation phải cùng contract, không throw lỗi ngoài contract.
ISP Các service import cả module repository lớn dù chỉ cần vài hàm. Tách repository/service interface nhỏ theo use case.
DIP Service phụ thuộc module cụ thể: mail, notification-events, repository object. Phụ thuộc vào port/function contract nhỏ; implementation cụ thể đặt ở composition layer hoặc service wrapper.

9. Hướng dẫn triển khai

Bước 1: Tạo facade để refactor ít ảnh hưởng

Nếu controller đang import * as boardingService from "./boarding.service.js", có thể giữ file boarding.service.ts làm facade re-export. Như vậy route/controller chưa cần đổi nhiều. 

// boarding.service.ts sau refactor giai đoạn đầu
export * from "./owner-boarding.service.js";
export * from "./staff-boarding.service.js";
export * from "./admin-boarding-room.service.js";

Bước 2: Tách mapper trước khi tách nghiệp vụ lớn

Tách mapper là bước ít rủi ro vì không đổi business rule. Chuyển các hàm map..., format..., get...Label ra file mapper/policy tương ứng, export và import lại.

Bước 3: Tách service theo role/use case

Với boarding, copy nguyên nhóm hàm owner sang owner-boarding.service.ts, staff sang staff-boarding.service.ts, admin room sang admin-boarding-room.service.ts. Sau mỗi lần tách, chạy TypeScript check.

Bước 4: Tách repository theo query ownership

Repository có thể tách sau service. Không nên vừa đổi query vừa đổi mapper cùng lúc. Mỗi repository mới chỉ chứa query phục vụ service tương ứng.

Bước 5: Tách policy/strategy cho rule thay đổi

Chỉ tạo Strategy/Factory cho rule có nhiều biến thể hoặc khả năng mở rộng rõ ràng: payment method, report export format, pricing, status transition, doctor assignment. Không tạo interface cho mọi service chỉ để có interface.

Bước 6: Chuẩn hóa controller

  • Controller không tự parse Zod nếu route đã validateRequest.
  • Controller dùng asyncHandler ở route, không cần try/catch lặp lại.
  • Controller dùng sendSuccess/sendPaginated nếu response đơn giản.
  • Không dùng (req as any).user; dùng req.user! theo shared express type.

10. Mẫu code refactor

Lưu ý khi dùng mẫu code: các snippet dưới đây là khung triển khai để tách trách nhiệm. Khi áp dụng vào code thật, phải giữ nguyên endpoint, status code và DTO shape hiện có. Đặc biệt với mapper, không được bỏ field đang được frontend sử dụng.

10.1. Mapper tách khỏi repository

// boarding.mapper.ts
import { AppError } from "../../shared/errors/app-error.js";
import { httpStatus } from "../../shared/errors/http-status.js";
import type {
  BoardingAlertLevel,
  BoardingInvoiceStatus,
  BoardingPaymentOption,
  BoardingRecordListItemDto,
  BoardingRecordListRow
} from "./boarding.types.js";

const boardingStatusLabels: Record<BoardingRecordListRow["boarding_status"], string> = {
  pending: "Chờ xác nhận",
  confirmed: "Chờ check-in",
  staying: "Đang lưu trú",
  checked_out: "Hoàn tất",
  cancelled: "Đã hủy",
  rejected: "Từ chối"
};

function getPaymentMethodLabel(paymentOption: BoardingPaymentOption | null): string {
  if (paymentOption === "online") return "Thanh toán online";
  if (paymentOption === "counter") return "Thanh toán tại quầy";
  return "Chưa cập nhật";
}

function getPaymentStatus(
  invoiceStatus: BoardingInvoiceStatus | null,
  hasSuccessPayment: boolean
): BoardingRecordListItemDto["payment"]["paymentStatus"] {
  if (invoiceStatus === "paid" || hasSuccessPayment) return "paid";
  if (invoiceStatus === "refunded") return "refunded";
  if (invoiceStatus === "cancelled") return "cancelled";
  return "unpaid";
}

function getPaymentStatusLabel(paymentStatus: BoardingRecordListItemDto["payment"]["paymentStatus"]): string {
  const labels: Record<BoardingRecordListItemDto["payment"]["paymentStatus"], string> = {
    paid: "Đã thanh toán",
    unpaid: "Chưa thanh toán",
    refunded: "Đã hoàn tiền",
    cancelled: "Đã hủy"
  };
  return labels[paymentStatus];
}

function getHealthStatusLabel(healthStatus: BoardingAlertLevel | "unknown"): string {
  const labels: Record<BoardingAlertLevel | "unknown", string> = {
    normal: "Bình thường",
    attention: "Cần chú ý",
    urgent: "Cần xử lý",
    unknown: "Chưa cập nhật"
  };
  return labels[healthStatus];
}

function normalizeTimestamp(value: unknown): string | null {
  if (!value) return null;
  if (value instanceof Date) return value.toISOString();
  return new Date(String(value)).toISOString();
}

function requireTimestamp(value: unknown): string {
  const normalized = normalizeTimestamp(value);
  if (!normalized) {
    throw new AppError("Dữ liệu thời gian lưu trú không hợp lệ", "INVALID_BOARDING_TIME", httpStatus.INTERNAL_SERVER_ERROR);
  }
  return normalized;
}

export function mapBoardingRecordListItem(row: BoardingRecordListRow): BoardingRecordListItemDto {
  const paymentStatus = getPaymentStatus(row.invoice_status, row.has_success_payment);
  const healthStatus: BoardingAlertLevel | "unknown" = row.alert_level ?? "unknown";
  const activeCare =
    row.boarding_status === "staying"
      ? {
        healthStatus,
        healthStatusLabel: getHealthStatusLabel(healthStatus),
        lastUpdatedAt: normalizeTimestamp(row.last_update_at)
      }
      : undefined;

  return {
    boardingRecordId: row.boarding_record_id,
    boardingCode: row.boarding_record_id,
    pet: {
      petId: row.pet_id,
      petName: row.pet_name,
      profileImageUrl: row.profile_image_url
    },
    room: {
      roomTypeId: row.room_type_id,
      roomTypeName: row.room_type_name
    },
    status: row.boarding_status,
    statusLabel: boardingStatusLabels[row.boarding_status],
    plannedCheckInAt: requireTimestamp(row.planned_check_in_at),
    plannedCheckOutAt: requireTimestamp(row.planned_check_out_at),
    plannedCheckInDate: row.planned_check_in_date,
    plannedCheckOutDate: row.planned_check_out_date,
    plannedDateRangeText: row.planned_date_range_text,
    stayDays: Number(row.stay_days),
    payment: {
      paymentOption: row.payment_option,
      paymentMethodLabel: getPaymentMethodLabel(row.payment_option),
      paymentStatus,
      paymentStatusLabel: getPaymentStatusLabel(paymentStatus)
    },
    estimatedTotal: Number(row.estimated_total),
    ...(activeCare ? { activeCare } : {})
  };
}

10.5. Pricing policy cho Grooming

// grooming-pricing.policy.ts
const largePetThresholdKg = 5;
const largePetSurchargeStepKg = 3;
const largePetSurchargeAmount = 50000;

export function calculateGroomingPrice(basePrice: number, weightKg: number): number {
  if (weightKg < largePetThresholdKg) return basePrice;

  const surchargeSteps = Math.floor((weightKg - largePetThresholdKg) / largePetSurchargeStepKg);
  return basePrice + surchargeSteps * largePetSurchargeAmount;
}

export function getAppliedPricingConditionLabel(weightKg: number): string {
  if (weightKg < largePetThresholdKg) return "Dưới 5kg";

  const surchargeSteps = Math.floor((weightKg - largePetThresholdKg) / largePetSurchargeStepKg);
  return surchargeSteps === 0 ? "Từ 5kg trở lên" : `Từ 5kg trở lên (+${surchargeSteps * largePetSurchargeAmount} VND)`;
}

10.6. Doctor assignment strategy

// doctor-assignment.strategy.ts
import type { AvailableDoctorRow } from "./appointments.types.js";

export interface DoctorAssignmentStrategy {
  selectDoctor(
    availableDoctors: AvailableDoctorRow[],
    excludedDoctorIds?: Set<string>,
    preferredDoctorId?: string | null
  ): AvailableDoctorRow | null;
}

export class LeastBusyDoctorAssignmentStrategy implements DoctorAssignmentStrategy {
  selectDoctor(
    availableDoctors: AvailableDoctorRow[],
    excludedDoctorIds = new Set<string>(),
    preferredDoctorId?: string | null
  ): AvailableDoctorRow | null {
    const candidates = availableDoctors.filter((doctor) => !excludedDoctorIds.has(doctor.user_id));
    if (preferredDoctorId) {
      const preferred = candidates.find((doctor) => doctor.user_id === preferredDoctorId);
      if (preferred) return preferred;
    }
    if (candidates.length === 0) return null;

    const minCount = Math.min(...candidates.map((doctor) => Number(doctor.confirmed_count_in_day)));
    return candidates.find((doctor) => Number(doctor.confirmed_count_in_day) === minCount) ?? null;
  }
}

10.8. Shared password service

// shared/security/password.service.ts
import { randomBytes, scrypt as scryptCallback, timingSafeEqual } from "node:crypto";
import { promisify } from "node:util";

const scrypt = promisify(scryptCallback);

export async function hashPassword(password: string): Promise<string> {
  const salt = randomBytes(16).toString("base64url");
  const derivedKey = (await scrypt(password, salt, 64)) as Buffer;
  return `scrypt$${salt}$${derivedKey.toString("base64url")}`;
}

export async function verifyPassword(password: string, passwordHash: string): Promise<boolean> {
  const [algorithm, salt, storedKey] = passwordHash.split("$");
  if (algorithm !== "scrypt" || !salt || !storedKey) return false;

  const derivedKey = (await scrypt(password, salt, 64)) as Buffer;
  const storedBuffer = Buffer.from(storedKey, "base64url");
  return storedBuffer.length === derivedKey.length && timingSafeEqual(storedBuffer, derivedKey);
}

export function generateTemporaryPassword(length = 10): string {
  const alphabet = "ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz23456789";
  const random = randomBytes(length);
  return Array.from(random, (byte) => alphabet[byte % alphabet.length]).join("");
}
